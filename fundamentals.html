影像基礎 (Image Fundamentals)

每一個量化結論都建立在「像素到底代表什麼」之上——物理尺度、位元深度、通道、檔案格式。讀錯這些，後續再厲害的演算法也會給你錯誤的生物學答案。

Every quantitative claim rests on what a pixel represents — physical size, bit depth, channels, file format. Get these wrong and the most sophisticated downstream algorithm will silently give you wrong biology.

概覽

為什麼從影像基礎開始？

顯微鏡輸出的不只是「圖片」，而是一個有物理尺度、動態範圍、通道與時間軸的多維數位陣列。生物影像分析最常見的錯誤都不是出在進階演算法，而是第一步讀檔時把通道順序、位元深度或體素各向異性搞錯——這種錯誤通常不會丟出例外，而是默默地給你錯誤答案。

本章建立你閱讀任何顯微影像所需的詞彙：從像素 / 體素、位元深度、通道與時間 (TCZYX)，到 OME-TIFF、CZI、LIF、ND2、OME-Zarr 等實際格式，以及 Bio-Formats 這個讀檔聖物。

A microscope does not output "a picture" — it outputs a multi-dimensional numeric array with physical scale, dynamic range, channels, and time. The most common errors in bioimage analysis are not in fancy algorithms; they are in step 1 — reading the file with the wrong channel order, bit depth, or voxel anisotropy. These errors do not throw exceptions; they silently return wrong answers.

This chapter gives you the vocabulary to read any microscopy file: pixels/voxels, bit depth, the TCZYX axis convention, OME-TIFF, CZI, LIF, ND2, and OME-Zarr, plus the holy reader library, Bio-Formats.

💡
核心原則：讀檔後第一件事永遠是 print(img.shape, img.dtype, img.dims, pixel_size)。十次有九次的 bug 在這行就現形。 Core principle: The first thing you do after reading a file is print(img.shape, img.dtype, img.dims, pixel_size). Nine out of ten bugs are caught by this line.
像素與體素

一、像素 vs. 體素與物理尺度

像素 (pixel) 是 2D 影像的最小單位；體素 (voxel) 是 3D 體積的最小單位。每一個像素 / 體素都有：

  • 數值：亮度（強度），可能是 uint8 / uint16 / float32 等型別
  • 物理大小：通常以 µm / pixel 或 nm / pixel 表示（pixel size, scale）
  • 位置：在影像座標系中的 (x, y) 或 (x, y, z)

各向異性 (Anisotropy)：共軛焦或光片顯微鏡的 Z 軸間距通常比 XY 大 3–10 倍（例如 0.2 × 0.2 × 1.0 µm）。如果不處理，3D 分割與形態學特徵會被嚴重扭曲。

A pixel is the smallest unit of a 2D image; a voxel is the smallest unit of a 3D volume. Each pixel/voxel has:

  • Value: intensity, typically uint8 / uint16 / float32
  • Physical size: µm/pixel or nm/pixel (pixel size, scale)
  • Position: (x, y) or (x, y, z) in image coordinates

Anisotropy: Confocal and light-sheet stacks usually have Z spacing 3–10× coarser than XY (e.g., 0.2 × 0.2 × 1.0 µm). Ignore it and your 3D segmentation and shape features will be badly distorted.

🔢

位元深度

uint8：0–255（一般 RGB 圖、JPEG）。
uint16：0–65535（多數科學相機，但常只用前 12 bits = 0–4095）。
float32：去噪 / 解卷積後的影像。
切忌「為了好看」把 16-bit 轉成 8-bit 後再分析。

uint8: 0–255 (consumer RGB, JPEG).
uint16: 0–65535 (most scientific cameras, often only the first 12 bits = 0–4095 are used).
float32: after denoising or deconvolution.
Never down-convert 16-bit to 8-bit "for display" then analyse.

🌈

通道

多個螢光標誌（DAPI、GFP、Cy5…）各佔一個通道。順序與分析語意密切相關——Mesmer 規定第 0 通道為核、第 1 通道為膜。讀錯通道順序，模型不會報錯，會默默給你錯誤分割。

Multiple fluorescent markers (DAPI, GFP, Cy5…) each occupy a channel. Order is semantically important — Mesmer requires channel 0 = nucleus, channel 1 = membrane. Wrong channel order yields wrong segmentation silently.

📐

軸序

OME 標準：TCZYX（時間、通道、Z、Y、X）。但 Fiji 預設 XYCZT、numpy 陣列預設 (Z, Y, X) 或 (Y, X, C)。讀檔後務必檢查 img.dims。

OME standard: TCZYX. But Fiji defaults to XYCZT, and numpy uses (Z, Y, X) or (Y, X, C). After loading, always check img.dims.

檔案格式

三、檔案格式：專有 vs. 開放

每家顯微鏡廠商都有自己的專有格式，用來保存原始像素與大量元資料（雷射功率、像素大小、Z step、stage 位置…）。專有格式有版本問題，所以分析時通常會轉換到開放格式。

Every microscope vendor has a proprietary format that stores raw pixels and rich metadata (laser power, pixel size, Z step, stage positions…). Proprietary formats have versioning issues, so analysis usually converts to an open format.

📁 專有格式

  • CZI (Zeiss ZEN)
  • LIF (Leica LAS X)
  • ND2 (Nikon NIS-Elements)
  • OIB / OIR (Olympus / Evident FluoView)
  • MRXS (Hamamatsu / 3DHistech 病理掃片)

📂 開放格式

  • TIFF / OME-TIFF：最通用，可載 OME-XML 元資料
  • HDF5 / N5：分塊存取，適合大資料 / BigDataViewer
  • OME-Zarr (NGFF)：雲端原生、多解析度金字塔，2025 年的新標準

程式碼

四、讀取顯微影像實作

# 1) Universal reader via bioio — handles CZI/LIF/ND2/OME-TIFF/OME-Zarr
from bioio import BioImage
img = BioImage("sample.czi")
print(img.shape, img.dims, img.dtype)        # (T, C, Z, Y, X), e.g. (1, 3, 41, 1024, 1024)
print(img.physical_pixel_sizes)               # PhysicalPixelSizes(Z=1.0, Y=0.21, X=0.21)  µm
print(img.channel_names)                      # ['DAPI', 'GFP', 'Cy5']

# Get one channel as numpy
nuc = img.get_image_data("ZYX", T=0, C=0)    # 3D nucleus channel

# 2) Fast OME-TIFF read with tifffile
import tifffile, numpy as np
arr = tifffile.imread("sample.ome.tif")        # shape e.g. (T,C,Z,Y,X) or (Y,X)
print(arr.shape, arr.dtype)

# 3) Cloud-native: OME-Zarr
import zarr
z = zarr.open("sample.ome.zarr", mode="r")
level0 = z[0]                                  # multi-scale level 0 (highest res)

# 4) WSI / pathology — use tiffslide (faster) or openslide
from tiffslide import TiffSlide
slide = TiffSlide("tumor.svs")
print(slide.level_dimensions, slide.properties["openslide.mpp-x"])
tile = slide.read_region((10000,10000), level=0, size=(512,512))
// Fiji macro — open any vendor format via Bio-Formats
run("Bio-Formats Importer", "open=[/path/sample.czi] color_mode=Default view=Hyperstack stack_order=XYCZT");

// Check pixel size
getPixelSize(unit, pw, ph, pd);
print("pixel size: " + pw + " x " + ph + " " + unit);
print("depth (Z step): " + pd);

// Get dimensions
getDimensions(w, h, ch, sl, fr);
print("W="+w+" H="+h+" C="+ch+" Z="+sl+" T="+fr);

// Save as OME-TIFF for downstream tools
run("Bio-Formats Exporter", "save=[/path/sample.ome.tif]");
⚠️
陷阱：把 16-bit 影像顯示「看起來太暗」就直接 img.astype(uint8) 是最常見的初學者錯誤——這會讓你失去 99% 的動態範圍。只在顯示時做 contrast stretch，分析請保留原本資料型別。 Pitfall: The most common beginner mistake is to convert a 16-bit image that "looks too dark" via img.astype(uint8) — that throws away 99% of the dynamic range. Contrast-stretch only for display; keep the native dtype for analysis.
最佳實踐

五、元資料與資料管理

影像分析的可重現性，10% 在演算法，90% 在元資料管理。一個 TIFF 沒寫 pixel size，整篇論文的所有 µm 都是猜的。

  • OME-XML：標準化元資料 schema，OME-TIFF 預設嵌入
  • BIDS-Microscopy：仿照 fMRI BIDS，制定影像資料的目錄結構
  • BioImage Archive：EMBL-EBI 的影像公共資料庫，類似 GEO/SRA 之於序列
  • IDR (Image Data Resource)：策展過的高品質公共影像資料集

Reproducibility in image analysis is 10% algorithm, 90% metadata management. A TIFF without pixel size means every µm in your paper is a guess.

  • OME-XML: standardized metadata schema, embedded in OME-TIFF
  • BIDS-Microscopy: BIDS-style directory layout for imaging
  • BioImage Archive: EMBL-EBI public image data repository (image counterpart of GEO/SRA)
  • IDR: curated, high-quality public image datasets
💡
實用建議：專案啟動時建一個 data_manifest.csv，每一列記錄一張影像、檔案路徑、物種、組織、染色、顯微鏡、像素大小、Z 間距、拍攝者、日期。這張表會在三年後救你的命。 Practical tip: At project start, create a data_manifest.csv — one row per image with path, species, tissue, staining, microscope, pixel size, Z step, operator, date. This table will save you three years from now.
