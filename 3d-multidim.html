3D 與多維分析

真實生物大多是 3D、多通道、有時間維度。3D 分割、各向異性處理、N-D 互動式檢視器、大資料流式運算——是 organoid、發育、light-sheet 工作流的基本門檻。

Most biology is 3D, multi-channel, and time-resolved. 3D segmentation, anisotropy handling, N-D viewers, and out-of-core computation are baseline for organoid, developmental, and light-sheet workflows.

開場故事 · STORY

從 1957 共軛焦到 2014 lattice light-sheet：4D 影像的崛起

1957 年 Marvin Minsky 申請共軛焦專利；1985 年第一張共軛焦 Z-stack 拍下；2014 年 Eric Betzig 在 Science 發表 lattice light-sheet，把單細胞動態的 4D imaging 推到 millisecond 與 min 解析度。

顯微鏡從 2D 切片進化到 5D (X, Y, Z, C, T)，但分析師仍經常忘記「Z 比 XY 大 5 倍」。各向異性、isotropic resampling、N-D viewer、dask out-of-core 是 3D 工作流的四大基石。

Marvin Minsky filed the confocal patent in 1957; the first confocal Z-stack came in 1985; Eric Betzig published lattice light-sheet in Science 2014, pushing single-cell 4-D imaging to millisecond and minute resolution.

Microscopy evolved from 2-D slices to 5-D (X, Y, Z, C, T) — yet analysts still forget that 'Z is 5× coarser than XY.' Anisotropy, isotropic resampling, N-D viewers, and dask out-of-core are the four cornerstones of 3-D workflows.

🤔 為什麼 confocal 與 light-sheet 的 anisotropy 處理方式不同？這背後反映兩種光學設計的根本差異。
🤔 Why do confocal and light-sheet handle anisotropy differently? The answer reflects two fundamental optical-design philosophies.
各向異性

一、各向異性體素

典型共軛焦：XY = 0.21 µm、Z = 1.0 µm。這意味一個體素在 Z 方向比 XY 大 5 倍。

後果：

  • 3D 距離計算 / 球度 / 表面積必須乘上 voxel size
  • 3D 分割若不告訴模型 anisotropy，cell 在 Z 軸會被誤判得「過短或過長」
  • 視覺化必須開「scale = (z, y, x)」才能看到正確的細胞比例

兩種處理策略：

  • (A) 傳給演算法 anisotropy（Cellpose 的 anisotropy=5.0）
  • (B) resample 到 isotropic（用 zoom / scipy zoom），memory 換正確性

Typical confocal: XY = 0.21 µm, Z = 1.0 µm — a voxel is 5× larger in Z than XY.

Consequences:

  • 3D distance / sphericity / surface area must be scaled by voxel size
  • 3D segmentation will mis-judge cell length in Z without anisotropy info
  • Visualization must use "scale = (z, y, x)" to show correct proportions

Two strategies:

  • (A) Pass anisotropy to the algorithm (Cellpose's anisotropy=5.0)
  • (B) Resample to isotropic (zoom / scipy zoom) — trades memory for correctness

二、3D 分割策略

策略作法優缺
真正 3D3D CNN 一次處理整 stack最準（密集組織）；最耗記憶體3D CNN processes whole stackMost accurate (dense tissue); memory-heavy

主流 3D 模型

  • Cellpose 3D：do_3D=True、anisotropy=5.0。一鍵啟動。
  • StarDist 3D：star-convex 多面體，圓核 3D 首選。
  • PlantSeg：植物 / 動物組織 3D 細胞邊界分割，CNN + graph partitioning。
  • nnU-Net 3D_fullres：自配置 3D U-Net；醫學影像 benchmark 標配。
  • Micro-SAM 3D：互動式 3D 分割，標一層即可向上下傳播。
大資料

三、處理大資料

Light-sheet 一次實驗常產生數百 GB 到 TB。一般 imread 整檔讀入 RAM 直接 OOM。解法是分塊存取與惰性運算。

  • OME-Zarr：多解析度金字塔的雲端原生格式（NGFF 標準），分塊 chunked。
  • N5 / HDF5：BigDataViewer 慣用，分塊隨機存取。
  • dask / dask-image：把陣列操作延後執行，scheduler 自動切塊與並行。
  • BigDataViewer / BigStitcher / mobie：TB 級互動視覺化。

A single light-sheet run often produces hundreds of GB to TB. Plain imread OOMs. Solution: chunked storage + lazy computation.

  • OME-Zarr: cloud-native pyramidal format (NGFF), chunked.
  • N5 / HDF5: BigDataViewer's preferred chunked storage.
  • dask / dask-image: defer execution; scheduler chunks + parallelizes automatically.
  • BigDataViewer / BigStitcher / mobie: TB-scale interactive visualization.
napari

四、napari

napari 是現代 Python bioimage 生態的互動中樞，原生支援 5D 影像、label layer、points / shapes / vectors / tracks / surfaces。豐富的 plugin 生態：

  • napari-cellpose、napari-stardist：分割
  • micro-sam：互動式提示分割
  • napari-bigstitcher：拼接
  • btrack / napari-arboretum：追蹤與譜系
  • napari-clemreg、napari-bigwarp：配準
  • napari-skimage-regionprops：互動量化

napari is the modern Python bioimage interactive hub — native 5D images, label/points/shapes/vectors/tracks/surfaces layers. Rich plugin ecosystem:

  • napari-cellpose, napari-stardist: segmentation
  • micro-sam: interactive promptable segmentation
  • napari-bigstitcher: stitching
  • btrack / napari-arboretum: tracking and lineage
  • napari-clemreg, napari-bigwarp: registration
  • napari-skimage-regionprops: interactive quantification

mdl = models.Cellpose(gpu=True, model_type="cyto3")
masks, _, _, _ = mdl.eval(
    stack,                    # [Z, Y, X] or [Z, Y, X, C]
    diameter=30,              # in XY pixels
    do_3D=True,              # true 3D
    anisotropy=5.0,           # Z is 5x XY voxel size
    channels=[0, 0]
)

# Alternative: 2D + stitch (faster, sparse cells)
masks2, _, _, _ = mdl.eval(stack, diameter=30, do_3D=False,
                           stitch_threshold=0.4)
# napari viewer + dask for large stacks
import napari, dask.array as da
from dask_image.imread import imread

# Lazy loading — no memory pressure
stack = imread("slices/*.tif")        # dask array
labels = imread("masks/*.tif")

viewer = napari.Viewer()
viewer.add_image(stack, scale=(5, 1, 1),   # (z, y, x) anisotropy
                 contrast_limits=[100, 4000])
viewer.add_labels(labels, scale=(5, 1, 1))

# Interactively segment one slice with cellpose-napari plugin
napari.run()
# Convert big TIFF to OME-Zarr (chunked pyramidal)
import tifffile, zarr
from ome_zarr.writer import write_image
from ome_zarr.io import parse_url

img = tifffile.memmap("huge.tif")               # memmap = lazy
store = parse_url("out.ome.zarr", mode="w").store
root = zarr.group(store=store)
write_image(image=img, group=root, axes="tczyx",
            storage_options={"chunks": (1,1,64,512,512)},
            scaler=None)

# Read back lazily
g = zarr.open("out.ome.zarr", mode="r")
arr = g[0]                                  # highest-resolution level
chunk = arr[0, 0, 10:12, 100:200, 100:200]   # tiny read

公式拆解

3D 體積與球度公式

Σ 逐步推導
V = voxel_count × dx × dy × dz  |  sphericity = π1/3 · (6V)2/3 / A
① 體積Volume V = 體素數 × dx · dy · dz。不傳 voxel size = 報告 voxel count 而非 μm³。V = voxel_count × dx · dy · dz. Without voxel size you're reporting counts, not μm³.
② 表面積Surface area 用 marching cubes 萃取 mesh，三角面積求和。Extract a mesh with marching cubes; sum triangle areas.
③ 球度Sphericity 完美球 = 1；不規則形 < 1。各向異性影響三維 sphericity。Perfect sphere = 1; irregular < 1. Anisotropy distorts 3-D sphericity.
④ 3D 距離3-D distance d = √((dx·Δx)² + (dy·Δy)² + (dz·Δz)²)；anisotropy 影響顯著。d = √((dx·Δx)² + (dy·Δy)² + (dz·Δz)²); anisotropy matters here.
推論：所有 3D 形態學特徵都「逃不過 voxel size」。要 cross-study 比較必須報告 voxel size 與 isotropic resampling 流程。
Implication: every 3-D morphology feature 'depends on voxel size.' Cross-study comparison requires reporting voxel size and any isotropic resampling.
思辨討論

3D 多維三大爭論

A. 為什麼 lattice LSFM 需要 multi-view fusion？
單視角光片解析度受限，6 視角融合後變 isotropic。背後的物理是？
Single-view LSFM has limited resolution; 6 views fused → isotropic. What's the physics?
提示：每個視角在「沿光片方向」最差。多視角覆蓋不同方向後，融合提升各方向解析度。
Hint: each view is worst 'along the sheet direction.' Multi-view covers different directions; fusion lifts all-directional resolution.
B. napari vs Fiji BigDataViewer 何時用哪個？
TB-scale 影像可視化你會選哪個？
For TB-scale visualization, which would you pick?
提示：BigDataViewer 仍是 Fiji 生態大資料王者；napari 在 Python + ML 整合場景更靈活。可並用。
Hint: BigDataViewer remains king in the Fiji big-data ecosystem; napari is more flexible for Python + ML. Use both.
C. PlantSeg 與 Cellpose 3D 哪個更通用？
「邊界式」(PlantSeg) vs 「中心式」(Cellpose) 在 3D 各有強項？
Boundary-based (PlantSeg) vs centre-based (Cellpose) — strengths?
提示：PlantSeg 對 cell-boundary 清楚的組織強（植物、上皮）；Cellpose 對形態異質、無明顯邊界的細胞 (immune cell, fibroblast) 強。
Hint: PlantSeg shines for tissues with clear boundaries (plants, epithelia); Cellpose shines for heterogeneous, edge-less cells (immune cells, fibroblasts).
精選案例

兩個 3D 多維傳奇

① Janelia LightSheet Pipeline
HHMI Janelia · lattice LSFM + dask + OME-Zarr

Janelia 的 lattice LSFM 一個小鼠胚胎 30 分鐘掃描產 1 TB 資料。Pipeline 完全採用 dask + OME-Zarr，沒有任何步驟整檔載入記憶體。從 raw → multi-view fusion → decon → 分割 → tracking 全部 lazy 處理。一張 4D 胚胎影像 14 個小時內處理完畢，是現代大資料 bioimage 的標竿。

Janelia's lattice LSFM produces 1 TB of data from a 30-minute mouse-embryo scan. The pipeline is fully dask + OME-Zarr, with no step loading the file whole. raw → multi-view fusion → decon → segmentation → tracking are all lazy. A 4-D embryo dataset is processed in under 14 hours — the modern big-data bioimage benchmark.

啟示：lazy 與分塊不是「奢侈」，是「必須」。
Lesson: lazy + chunked is not 'luxury' — it's necessary.
② PlantSeg：植物細胞邊界 SOTA
Wolny et al., eLife 9:e57613 (2020) · github.com/hci-unihd/plant-seg

Adrian Wolny 等 2020 在 eLife 發表 PlantSeg：DL 邊界檢測 + graph partitioning。在 Arabidopsis 根尖、葉表皮等 3D 共軛焦資料把細胞邊界切得比手工還準。EMBL Heidelberg 與 Yale 廣泛使用，植物器官 morphogenesis 研究的事實工具。後也被應用到動物上皮（如 Drosophila wing disc）。

Adrian Wolny et al. (eLife 2020) released PlantSeg: DL boundary detection + graph partitioning. Cells in Arabidopsis root tips and leaf epidermis are segmented more accurately than by hand. Widely used at EMBL Heidelberg and Yale — the de facto tool for plant-organ morphogenesis. Later extended to animal epithelia (e.g., Drosophila wing disc).

啟示：領域特化的工具仍有不可替代的價值，generalist 不會吃掉所有 niche。
Lesson: domain-specialized tools still have irreplaceable value; generalists won't eat every niche.
