影像基礎 (Image Fundamentals)

每一個量化結論都建立在「像素到底代表什麼」之上——物理尺度、位元深度、通道、檔案格式。讀錯這些，後續再厲害的演算法也會給你錯誤的生物學答案。

Every quantitative claim rests on what a pixel represents — physical size, bit depth, channels, file format. Get these wrong and the most sophisticated downstream algorithm will silently give you wrong biology.

開場故事 · STORY

一個像素能說多少生物學？

1873 年，蔡司公司的光學家 Ernst Abbe 寫下著名的繞射極限公式 d = λ/(2·NA)——告訴我們即使有完美的鏡頭，兩個點靠得太近就會「糊在一起」。一百五十年後，現代顯微鏡仍受這條公式束縛，但我們發明了更聰明的方法：用多張影像、多波長、多時間、多視角來重建生物的真相。

於是每一張顯微影像不再是「一張圖」，而是有著時間、通道、Z、Y、X 五個維度的多維陣列，外加雷射功率、像素大小、物鏡資訊等元資料。讀錯任何一個維度，你的「奈米級結論」其實可能是讀檔時把通道順序搞反了。

In 1873, Zeiss optician Ernst Abbe wrote down the famous diffraction limit d = λ/(2·NA), telling us that two points too close together will always "blur into one." A hundred and fifty years later, we are still bound by his formula — but we've invented smarter tricks: multi-image, multi-wavelength, multi-time, multi-view reconstruction.

So a microscopy image is no longer "a picture" but a 5-D array with time, channels, Z, Y, X — plus laser power, pixel size, objective info. Read any dimension wrong and your "nanometre conclusion" might just be a channel swap on disk.

🤔 想一想：如果同一張 confocal Z-stack 在 ImageJ 看是 (T,Z,C,Y,X)，在 numpy 看是 (Z,Y,X,C)，誰才對？答案是——兩個都對，因為它們是同一個 5D 陣列的不同投影方式。重點在 你 必須知道自己在哪個視角。
🤔 Think about it: if the same confocal Z-stack reads as (T,Z,C,Y,X) in ImageJ but (Z,Y,X,C) in numpy, which is "correct"? — Both, actually. They're different projections of the same 5-D array. What matters is that you know which view you're in.
概覽

為什麼從影像基礎開始？

顯微鏡輸出的不只是「圖片」，而是一個有物理尺度、動態範圍、通道與時間軸的多維數位陣列。生物影像分析最常見的錯誤都不是出在進階演算法，而是第一步讀檔時把通道順序、位元深度或體素各向異性搞錯——這種錯誤通常不會丟出例外，而是默默地給你錯誤答案。

本章建立你閱讀任何顯微影像所需的詞彙：從像素 / 體素、位元深度、通道與時間 (TCZYX)，到 OME-TIFF、CZI、LIF、ND2、OME-Zarr 等實際格式，以及 Bio-Formats 這個讀檔聖物。

A microscope does not output "a picture" — it outputs a multi-dimensional numeric array with physical scale, dynamic range, channels, and time. The most common errors in bioimage analysis are not in fancy algorithms; they are in step 1 — reading the file with the wrong channel order, bit depth, or voxel anisotropy. These errors do not throw exceptions; they silently return wrong answers.

This chapter gives you the vocabulary to read any microscopy file: pixels/voxels, bit depth, the TCZYX axis convention, OME-TIFF, CZI, LIF, ND2, and OME-Zarr, plus the holy reader library, Bio-Formats.

💡
核心原則：讀檔後第一件事永遠是 print(img.shape, img.dtype, img.dims, pixel_size)。十次有九次的 bug 在這行就現形。 Core principle: The first thing you do after reading a file is print(img.shape, img.dtype, img.dims, pixel_size). Nine out of ten bugs are caught by this line.
像素與體素

一、像素 vs. 體素與物理尺度

像素 (pixel) 是 2D 影像的最小單位；體素 (voxel) 是 3D 體積的最小單位。每一個像素 / 體素都有：

  • 數值：亮度（強度），可能是 uint8 / uint16 / float32 等型別
  • 物理大小：通常以 µm / pixel 或 nm / pixel 表示（pixel size, scale）
  • 位置：在影像座標系中的 (x, y) 或 (x, y, z)

各向異性 (Anisotropy)：共軛焦或光片顯微鏡的 Z 軸間距通常比 XY 大 3–10 倍（例如 0.2 × 0.2 × 1.0 µm）。如果不處理，3D 分割與形態學特徵會被嚴重扭曲。

A pixel is the smallest unit of a 2D image; a voxel is the smallest unit of a 3D volume. Each pixel/voxel has:

  • Value: intensity, typically uint8 / uint16 / float32
  • Physical size: µm/pixel or nm/pixel (pixel size, scale)
  • Position: (x, y) or (x, y, z) in image coordinates

Anisotropy: Confocal and light-sheet stacks usually have Z spacing 3–10× coarser than XY (e.g., 0.2 × 0.2 × 1.0 µm). Ignore it and your 3D segmentation and shape features will be badly distorted.

🔢

位元深度

uint8：0–255（一般 RGB 圖、JPEG）。
uint16：0–65535（多數科學相機，但常只用前 12 bits = 0–4095）。
float32：去噪 / 解卷積後的影像。
切忌「為了好看」把 16-bit 轉成 8-bit 後再分析。

uint8: 0–255 (consumer RGB, JPEG).
uint16: 0–65535 (most scientific cameras, often only the first 12 bits = 0–4095 are used).
float32: after denoising or deconvolution.
Never down-convert 16-bit to 8-bit "for display" then analyse.

🌈

通道

多個螢光標誌（DAPI、GFP、Cy5…）各佔一個通道。順序與分析語意密切相關——Mesmer 規定第 0 通道為核、第 1 通道為膜。讀錯通道順序，模型不會報錯，會默默給你錯誤分割。

Multiple fluorescent markers (DAPI, GFP, Cy5…) each occupy a channel. Order is semantically important — Mesmer requires channel 0 = nucleus, channel 1 = membrane. Wrong channel order yields wrong segmentation silently.

📐

軸序

OME 標準：TCZYX（時間、通道、Z、Y、X）。但 Fiji 預設 XYCZT、numpy 陣列預設 (Z, Y, X) 或 (Y, X, C)。讀檔後務必檢查 img.dims。

OME standard: TCZYX. But Fiji defaults to XYCZT, and numpy uses (Z, Y, X) or (Y, X, C). After loading, always check img.dims.

檔案格式

三、檔案格式：專有 vs. 開放

每家顯微鏡廠商都有自己的專有格式，用來保存原始像素與大量元資料（雷射功率、像素大小、Z step、stage 位置…）。專有格式有版本問題，所以分析時通常會轉換到開放格式。

Every microscope vendor has a proprietary format that stores raw pixels and rich metadata (laser power, pixel size, Z step, stage positions…). Proprietary formats have versioning issues, so analysis usually converts to an open format.

📁 專有格式

  • CZI (Zeiss ZEN)
  • LIF (Leica LAS X)
  • ND2 (Nikon NIS-Elements)
  • OIB / OIR (Olympus / Evident FluoView)
  • MRXS (Hamamatsu / 3DHistech 病理掃片)

📂 開放格式

  • TIFF / OME-TIFF：最通用，可載 OME-XML 元資料
  • HDF5 / N5：分塊存取，適合大資料 / BigDataViewer
  • OME-Zarr (NGFF)：雲端原生、多解析度金字塔，2025 年的新標準

程式碼

四、讀取顯微影像實作

# 1) Universal reader via bioio — handles CZI/LIF/ND2/OME-TIFF/OME-Zarr
from bioio import BioImage
img = BioImage("sample.czi")
print(img.shape, img.dims, img.dtype)        # (T, C, Z, Y, X), e.g. (1, 3, 41, 1024, 1024)
print(img.physical_pixel_sizes)               # PhysicalPixelSizes(Z=1.0, Y=0.21, X=0.21)  µm
print(img.channel_names)                      # ['DAPI', 'GFP', 'Cy5']

# Get one channel as numpy
nuc = img.get_image_data("ZYX", T=0, C=0)    # 3D nucleus channel

# 2) Fast OME-TIFF read with tifffile
import tifffile, numpy as np
arr = tifffile.imread("sample.ome.tif")        # shape e.g. (T,C,Z,Y,X) or (Y,X)
print(arr.shape, arr.dtype)

# 3) Cloud-native: OME-Zarr
import zarr
z = zarr.open("sample.ome.zarr", mode="r")
level0 = z[0]                                  # multi-scale level 0 (highest res)

# 4) WSI / pathology — use tiffslide (faster) or openslide
from tiffslide import TiffSlide
slide = TiffSlide("tumor.svs")
print(slide.level_dimensions, slide.properties["openslide.mpp-x"])
tile = slide.read_region((10000,10000), level=0, size=(512,512))
// Fiji macro — open any vendor format via Bio-Formats
run("Bio-Formats Importer", "open=[/path/sample.czi] color_mode=Default view=Hyperstack stack_order=XYCZT");

// Check pixel size
getPixelSize(unit, pw, ph, pd);
print("pixel size: " + pw + " x " + ph + " " + unit);
print("depth (Z step): " + pd);

// Get dimensions
getDimensions(w, h, ch, sl, fr);
print("W="+w+" H="+h+" C="+ch+" Z="+sl+" T="+fr);

// Save as OME-TIFF for downstream tools
run("Bio-Formats Exporter", "save=[/path/sample.ome.tif]");
⚠️
陷阱：把 16-bit 影像顯示「看起來太暗」就直接 img.astype(uint8) 是最常見的初學者錯誤——這會讓你失去 99% 的動態範圍。只在顯示時做 contrast stretch，分析請保留原本資料型別。 Pitfall: The most common beginner mistake is to convert a 16-bit image that "looks too dark" via img.astype(uint8) — that throws away 99% of the dynamic range. Contrast-stretch only for display; keep the native dtype for analysis.
最佳實踐

五、元資料與資料管理

影像分析的可重現性，10% 在演算法，90% 在元資料管理。一個 TIFF 沒寫 pixel size，整篇論文的所有 µm 都是猜的。

  • OME-XML：標準化元資料 schema，OME-TIFF 預設嵌入
  • BIDS-Microscopy：仿照 fMRI BIDS，制定影像資料的目錄結構
  • BioImage Archive：EMBL-EBI 的影像公共資料庫，類似 GEO/SRA 之於序列
  • IDR (Image Data Resource)：策展過的高品質公共影像資料集

Reproducibility in image analysis is 10% algorithm, 90% metadata management. A TIFF without pixel size means every µm in your paper is a guess.

  • OME-XML: standardized metadata schema, embedded in OME-TIFF
  • BIDS-Microscopy: BIDS-style directory layout for imaging
  • BioImage Archive: EMBL-EBI public image data repository (image counterpart of GEO/SRA)
  • IDR: curated, high-quality public image datasets
💡
實用建議：專案啟動時建一個 data_manifest.csv，每一列記錄一張影像、檔案路徑、物種、組織、染色、顯微鏡、像素大小、Z 間距、拍攝者、日期。這張表會在三年後救你的命。 Practical tip: At project start, create a data_manifest.csv — one row per image with path, species, tissue, staining, microscope, pixel size, Z step, operator, date. This table will save you three years from now.

公式拆解

Abbe 與 Nyquist：兩個改變一切的公式

Σ 逐步推導
dAbbe = λ / (2 · NA)
① λ ：螢光發射波長。DAPI ≈ 461 nm, GFP ≈ 510 nm, Cy5 ≈ 670 nm。短波長 → 更高解析度（這就是為什麼 STORM 喜歡用 405 雷射）。: emission wavelength. DAPI ≈ 461 nm, GFP ≈ 510 nm, Cy5 ≈ 670 nm. Shorter λ → higher resolution (why STORM loves 405 nm lasers).
② NA ：數值孔徑 = n·sin θ。氣鏡 NA ≤ 0.95；油鏡 NA 可達 1.4–1.49（n=1.515）。NA 加倍 → 解析度減半（更好）。: numerical aperture = n·sin θ. Air ≤ 0.95; oil reaches 1.4–1.49 (n=1.515). Doubling NA halves d (better).
③ 範例Example ：GFP + 60× 油鏡 → d = 510 / (2 × 1.4) ≈ 182 nm。這是 widefield 的解析度極限。: GFP + 60× oil → d = 510 / (2 × 1.4) ≈ 182 nm. The widefield limit.
pixel size ≤ dAbbe / 2.3   (Nyquist)
④ Nyquist ：要保留訊息，像素必須夠細。182 / 2.3 ≈ 79 nm/px。比 d_Abbe 大 → aliasing；比 d_Abbe 小太多 → 過取樣浪費 SNR。: pixels must be small enough. 182 / 2.3 ≈ 79 nm/px. Larger than d_Abbe → aliasing; much smaller → wastes SNR with oversampling.
所以 confocal 60× 油鏡 GFP 通道的像素大小設 ~80 nm/px 是 Nyquist 黃金值。低於這個值的「高放大率」其實是空轉，只會降低 SNR 而沒有取得新訊息。
So ~80 nm/px is the Nyquist sweet spot for confocal 60× oil GFP. "Higher magnification" below this is empty zoom — lowers SNR without adding information.

三個引人深思的問題

A. 8-bit 真的「夠用」嗎？
你的螢光定量論文圖片在投稿前被審稿人問：「為什麼把 16-bit 原圖轉成 8-bit 顯示？是不是犧牲了動態範圍？」你會怎麼回答？
A reviewer asks: "Why did you display your 16-bit data as 8-bit? Did you sacrifice dynamic range?" How do you respond?
提示：顯示 ≠ 分析。先確認所有定量步驟都在 16-bit 上完成；顯示用 8-bit 配 percentile-based contrast（如 0.1–99.9%）是普遍實踐，記得在 methods 寫明。
Hint: display ≠ analysis. Confirm all quantitation happened in 16-bit; 8-bit display with percentile contrast (0.1–99.9%) is standard. Document it in methods.
B. Pixel size 為何決定論文結論？
同一張 confocal 影像，A 實驗室寫 0.21 μm/px、B 寫 0.42 μm/px。後續測量的「細胞直徑」會差多少？這對 cell-cycle 統計推論影響為何？
Two labs report the same confocal image as 0.21 μm/px vs. 0.42 μm/px. How different are their "cell diameters"? What does that do to cell-cycle statistics?
提示：所有以 μm 表示的形態值都會差 2×。建議：每篇論文必須記錄 acquisition log 與 ome-tiff metadata 雙保險，並在 methods 寫出 pixel size。
Hint: every micron-based morphology value would differ by 2×. Record both acquisition logs and OME-TIFF metadata, and report pixel size in methods.
C. OME-Zarr 會淘汰 TIFF 嗎？
你的單位仍用 TIFF 為主，但同事推薦 OME-Zarr。在何種情境下換格式的「邊際效益」最高？哪些情境暫時不必？
Your lab still uses TIFF, but a colleague pushes OME-Zarr. When does switching pay off most? When is it not worth it yet?
提示：>5–10 GB 與 cloud / remote 場景受益最大（可只下載 ROI 而非整檔）；<1 GB 單檔案桌面工作流，TIFF 仍很合理。
Hint: >5–10 GB or cloud workflows gain most (stream ROIs without downloading the file). For <1 GB local workflows, TIFF is still reasonable.
精選案例

兩個影響深遠的故事

① Bio-Formats 如何拯救了顯微影像分析
Linkert et al., J. Cell Biol. 2010 · DOI: 10.1083/jcb.201004104 · 1,000+ citations

2000 年代初，每家顯微鏡廠商都用自家格式（CZI、LIF、ND2、OIB…），分析者必須轉檔、丟元資料。OME (Open Microscopy Environment) 在 2006 年發布 Bio-Formats Java library，支援 ~150 種廠商格式並保留 OME-XML metadata，徹底改變生態：Fiji、QuPath、CellProfiler、napari 全部建立在它之上。如今 Bio-Formats 被引用超過 1,000 次，是顯微影像分析的「水電」。

In the early 2000s, every vendor used its own format (CZI, LIF, ND2, OIB…) — analysts had to convert and lose metadata. OME released Bio-Formats in 2006 — a Java library supporting ~150 vendor formats while preserving OME-XML metadata — and changed everything. Fiji, QuPath, CellProfiler, napari all build on it. Bio-Formats has >1,000 citations and is microscopy's plumbing.

啟示：標準化看似乏味，卻是讓整個社群可以累積知識的關鍵基礎建設。
Lesson: standardization looks boring but is the infrastructure that lets a community accumulate knowledge.
② Allen Cell Explorer 的 6 TB 細胞地圖
Allen Institute for Cell Science · allencell.org · OME-Zarr on AWS S3

Allen Institute 公開了 200,000+ 張人類 iPSC 的 3D 共軛焦影像 + AI 推論結果（每個胞器一個 channel），整體 6 TB。為了讓全球研究者能在瀏覽器即時瀏覽，他們全部轉成 OME-Zarr 並架在 S3 上。任何人 napari --plugin napari-ome-zarr "s3://..." 就能秒開。這正是 OME-Zarr 設計初衷的最佳範例。

The Allen Institute publishes 200,000+ 3-D confocal stacks of human iPSCs with per-organelle AI inference channels — 6 TB total. To let anyone stream them in a browser, they converted everything to OME-Zarr on S3. napari --plugin napari-ome-zarr "s3://..." opens any cell in seconds. The poster child for cloud-native bioimage data.

啟示：當資料大到無法下載，「不必下載」的格式設計才是解方。
Lesson: when data is too big to download, the right answer is a "no download needed" format design.
常見陷阱

影像基礎六大陷阱

P1通道順序顛倒
Mesmer 規定 ch0=核、ch1=膜；順序錯不會報錯，但分割完全錯誤。
Mesmer requires ch0=nucleus, ch1=membrane; swapping silently breaks segmentation.
讀檔後立刻列印 channel_names 並肉眼對照預期。
Always print channel_names and visually verify right after loading.
P2無腦 astype(uint8)
把 16-bit 直接降成 8-bit，99% 動態範圍消失。
Casting 16-bit to 8-bit destroys 99% of dynamic range.
分析保留原 dtype；顯示用 percentile contrast stretch。
Keep native dtype for analysis; use percentile contrast for display.
P3忽略各向異性
Z spacing 比 XY 大 5× 卻不告訴模型 → 3D 形狀「壓扁」。
Z is 5× coarser than XY but you don't say so → 3-D cells get "flattened."
傳 anisotropy=5 給 segmenter 或先 resample 到 isotropic。
Pass anisotropy=5 to the segmenter or resample to isotropic first.
P4WSI 用 tifffile 整檔載入
.svs / .ndpi 是 giga-pixel 金字塔，整檔讀入直接 OOM。
.svs / .ndpi are gigapixel pyramids; loading whole = OOM.
改用 OpenSlide / TiffSlide tile 級存取。
Use OpenSlide / TiffSlide tile-level access.
P5不記錄 pixel size
TIFF 沒寫 μm/px，論文裡所有 μm 數字都是猜的。
TIFF lacking μm/px → every "μm" in the paper is a guess.
儲存時保留 OME-XML metadata；用 OME-TIFF。
Preserve OME-XML metadata; save as OME-TIFF.
P6過取樣假高解析
像素大小遠小於 d_Abbe/2.3，看起來「更細」但其實只有 SNR 變差。
Pixel ≪ d_Abbe/2.3 "looks sharper" but is just worse SNR — empty zoom.
先算 Nyquist 上限再設 acquisition pixel size。
Compute the Nyquist limit and set acquisition pixel size to match.
